# 📋 코드 리뷰 가이드라인

## 🌟 코드 리뷰의 목적

코드 리뷰는 코드의 품질을 향상시키고, 잠재적인 버그를 조기에 발견하며, 지식을 공유하고 팀 전체의 개발 역량을 강화하는 데 중요한 과정입니다. 단순히 오류를 찾아내는 것을 넘어, 더 나은 코드와 더 나은 개발자를 만드는 데 기여합니다.

## 🔑 주요 검토 사항

코드 리뷰 시 다음 영역에 집중하여 검토하면 효과적입니다.

### 1. 기능 (Functionality)
- **요구사항 충족**: 코드가 원래의 요구사항을 정확히 구현하는가?
- **예외 처리**: 모든 가능한 엣지 케이스와 오류 상황이 적절하게 처리되는가?
- **예상치 못한 동작**: 코드가 예상치 못한 부작용이나 오작동을 일으키지 않는가?

### 2. 가독성 및 유지보수성 (Readability & Maintainability)
- **명확성**: 코드를 읽고 이해하기 쉬운가? 변수명, 함수명 등이 목적을 명확히 설명하는가?
- **일관성**: 프로젝트의 코딩 컨벤션(스타일, 포맷팅, 네이밍)을 따르는가?
- **모듈화**: 코드가 응집도 높고 결합도 낮게 잘 분리되어 있는가? (단일 책임 원칙)
- **중복**: 불필요한 코드 중복은 없는가?
- **주석**: 복잡하거나 중요한 로직에 대한 주석이 적절하게 작성되어 있는가? (What보다는 Why)

### 3. 성능 (Performance)
- **효율성**: 비효율적인 알고리즘이나 데이터 구조를 사용하지 않는가?
- **자원 사용**: 불필요한 리소스 낭비(메모리, CPU)는 없는가?
- **최적화**: 성능 개선 여지가 있는 부분이 있는가? (과도한 최적화는 지양)

### 4. 보안 (Security)
- **취약점**: SQL Injection, XSS, 경로 조작 등 일반적인 보안 취약점에 노출될 가능성은 없는가?
- **민감 정보**: API 키, 비밀번호 등 민감 정보가 하드코딩되거나 로그에 기록되지 않는가?
- **접근 제어**: 권한이 없는 사용자가 중요 기능에 접근할 수 없는가?

### 5. 테스트 (Tests)
- **테스트 존재 여부**: 새로 추가되거나 변경된 코드에 대한 테스트 코드가 작성되었는가?
- **테스트 품질**: 테스트가 충분히 포괄적이며, 정확하게 작동하는가? (단위 테스트, 통합 테스트)
- **테스트 가독성**: 테스트 코드를 읽고 이해하기 쉬운가?

### 6. 설계 및 아키텍처 (Design & Architecture)
- **설계 의도**: 기존 시스템 설계 원칙과 아키텍처에 부합하는가?
- **확장성**: 향후 기능 확장을 고려하여 설계되었는가?
- **재사용성**: 다른 곳에서 재사용될 수 있는 컴포넌트가 있는가?

## 🧑‍💻 리뷰어를 위한 베스트 프랙티스

- **친절하고 건설적인 피드백**: 코드 작성자의 노력에 감사하고, 비난보다는 개선을 위한 제안을 하세요. "이 코드는 틀렸다" 대신 "이렇게 변경하면 더 좋을 것 같아요. 왜냐하면..."과 같이 제안하세요.
- **명확하고 구체적인 지적**: "코드가 복잡해요" 대신 "이 `for` 루프와 `if` 조건문을 `filter()` 함수로 바꾸면 더 간결해질 것 같아요"와 같이 구체적인 대안을 제시하세요.
- **질문 활용**: 때로는 직접적인 지적보다 "이 부분은 왜 이렇게 구현했나요?", "이 경우에 어떻게 동작할까요?"와 같이 질문하는 것이 더 나은 학습을 유도합니다.
- **중요도 표시**: 피드백의 중요도(필수, 권장, 선택사항)를 표시하면 작성자가 우선순위를 정하는 데 도움이 됩니다.
- **모든 변경사항 리뷰**: 작은 수정이라도 놓치지 않고 모든 변경사항을 꼼꼼히 확인하세요.
- **적절한 시간 투자**: 충분한 시간을 들여 코드를 이해하고 심층적으로 검토하세요.

## ✍️ 코드 작성자를 위한 베스트 프랙티스

- **리뷰 요청 전 자체 검토**: 푸시하기 전에 스스로 자신의 코드를 리뷰하며 오류나 개선 사항을 먼저 찾아보세요.
- **변경 사항 요약**: PR(Pull Request) 또는 커밋 메시지에 변경된 내용과 그 이유를 명확하게 설명하세요. 리뷰어가 코드를 이해하는 데 큰 도움이 됩니다.
- **피드백에 대한 감사**: 리뷰어의 시간과 노력에 감사하며 모든 피드백을 열린 마음으로 받아들이세요.
- **논쟁보다는 이해**: 의견 차이가 있을 경우, 감정적으로 대응하기보다 상대방의 관점을 이해하려고 노력하고 합리적으로 논의하세요.
- **모든 피드백 반영 또는 설명**: 모든 피드백에 대해 반영했거나, 반영하지 않은 이유를 명확하게 설명하세요.
- **작은 단위로 PR 생성**: 한번에 많은 코드를 리뷰하는 것은 어렵습니다. 가능하면 기능을 작은 단위로 나누어 PR을 생성하세요.

**효율적인 코드 리뷰는 팀의 성장을 촉진하고, 소프트웨어 품질을 높이는 핵심적인 과정입니다.**
모두 적극적으로 참여하여 더 나은 코드와 문화를 만들어 나갑시다.
