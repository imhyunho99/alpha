# 🕵️‍♂️ Alpha Project 상세 코드 리뷰

이 문서는 Alpha 프로젝트의 주요 소스 코드 파일들을 상세하게 분석한 리뷰 보고서입니다. 각 파일의 역할, 주요 로직, 그리고 발견된 문제점과 개선 방안을 구체적으로 기술합니다.

---

## 1. 🐍 `alpha_server/main.py`

### 📄 파일 개요
FastAPI 프레임워크를 사용한 백엔드 서버의 진입점입니다. 클라이언트(GUI)로부터 요청을 받아 데이터를 처리하고 결과를 반환합니다.

### 🔍 주요 코드 분석

- **`progress_status` (전역 변수)**
  - **코드**: `progress_status = {}`
  - **분석**: 백그라운드 작업(데이터 업데이트, 모델 학습)의 진행률을 저장하는 데 사용됩니다.
  - **문제점**:
    - **동시성 문제**: 여러 요청이 동시에 들어올 경우 데이터 경합(Race Condition)이 발생할 수 있습니다.
    - **확장성 불가**: 서버를 여러 프로세스(Gunicorn 워커 등)로 띄우면 메모리가 공유되지 않아 진행률 정보가 유실됩니다.
  - **개선안**: Redis 같은 외부 저장소나 데이터베이스를 사용하여 상태를 관리해야 합니다.

- **`assess_portfolio` (엔드포인트)**
  - **코드**: 포트폴리오 내의 각 자산에 대해 반복문(`for holding in portfolio.holdings`)을 돌며 `get_latest_price` 등을 호출합니다.
  - **분석**: 동기적(Synchronous)으로 작성되어 있어, 자산 개수가 많아질수록 응답 시간이 선형적으로 증가합니다.
  - **문제점**: I/O 바운드 작업(DB 조회, 연산)이 메인 스레드를 차단(Block)하여 다른 요청 처리를 지연시킵니다.
  - **개선안**: `async def`와 `await`, 그리고 `asyncio.gather`를 사용하여 병렬로 처리하거나, 배치(Batch) 쿼리로 한 번에 조회하도록 리팩토링해야 합니다.

## 2. 💾 `alpha_server/data_handler.py`

### 📄 파일 개요
QuestDB 및 yfinance와 연동하여 시장 데이터를 수집, 저장, 조회하는 핵심 모듈입니다.

### 🔍 주요 코드 분석

- **`load_data_from_questdb` (함수)**
  - **코드**: `query = f"SELECT * FROM market_data WHERE ticker = '{ticker}' ..."`
  - **🚨 치명적 문제점 (Critical)**: **SQL Injection 취약점**이 존재합니다. 사용자 입력(ticker)이 검증 없이 SQL 쿼리 문자열에 직접 삽입됩니다.
  - **개선안**: 반드시 파라미터화된 쿼리(Parameterized Query)를 사용해야 합니다. (예: `execute("... WHERE ticker = ?", (ticker,))`)

- **`insert_data_to_db` (함수)**
  - **코드**: DataFrame의 각 행(row)을 반복하며 `INSERT` 쿼리를 실행합니다.
  - **문제점**: **성능 저하**의 주원인입니다. 수천, 수만 건의 데이터를 한 건씩 넣는 것은 매우 비효율적입니다.
  - **개선안**: 대량 삽입(Bulk Insert) 기능을 사용하거나, QuestDB의 ILP(Influx Line Protocol) 인터페이스를 활용하여 고속으로 데이터를 적재해야 합니다.

- **`update_all_data` (함수)**
  - **로직**: 기존 데이터를 고려하지 않고 전체 기간의 데이터를 다시 다운로드하는 것으로 보입니다.
  - **문제점**: 불필요한 네트워크 트래픽과 시간 낭비가 발생합니다.
  - **개선안**: DB에 저장된 마지막 날짜를 조회하고, 그 이후의 데이터만 `yfinance`에서 가져오는 **증분 업데이트(Incremental Update)** 방식으로 변경해야 합니다.

## 3. 🧠 `alpha_server/model_handler.py`

### 📄 파일 개요
머신러닝 모델(RandomForest 등)을 학습하고, 예측을 수행하며, 모델 파일을 관리합니다.

### 🔍 주요 코드 분석

- **특성 엔지니어링 (Feature Engineering) 로직**
  - **코드**: `train_model` 함수와 `predict_latest` 함수 양쪽에 이동평균(SMA), RSI 등을 계산하는 코드가 중복되어 있습니다.
  - **문제점**: **코드 중복(Duplication)**. 로직을 수정할 때 두 곳을 모두 고쳐야 하며, 실수로 하나를 누락하면 학습 때와 예측 때의 데이터가 달라져 성능이 급격히 저하됩니다.
  - **개선안**: `create_features(df)`라는 별도 함수로 로직을 분리하여 재사용해야 합니다.

- **`joblib.load` (함수)**
  - **코드**: 저장된 모델 파일(`.joblib`)을 로드합니다.
  - **보안 이슈**: `joblib`이나 `pickle`은 신뢰할 수 없는 데이터를 로드할 때 **임의 코드 실행(RCE)** 취약점이 있습니다.
  - **개선안**: 모델 파일 저장 경로에 대한 권한을 엄격히 관리하거나, ONNX 같은 더 안전한 포맷 사용을 고려해볼 수 있습니다. (현재로서는 내부 생성 파일만 로드하므로 위험도는 중간 수준입니다.)

## 4. ⚖️ `alpha_server/ensemble_model.py`

### 📄 파일 개요
여러 모델의 예측 결과를 종합하여 최종 판단을 내리는 앙상블 로직을 담당합니다.

### 🔍 주요 코드 분석

- **`ensemble_predict` (함수)**
  - **장점**: 개별 모델의 오류를 잘 처리하고(try-except), 가중치(weights)를 설정 파일이나 변수로 관리하기 쉽게 설계되어 구조적으로 우수합니다.
  - **코드**: `results = {'rf': ..., 'xgb': ..., 'lgbm': ...}`
  - **분석**: 각 모델의 결과를 딕셔너리로 모으고 소프트 보팅(Soft Voting) 등을 수행합니다.

- **`batch_ensemble_predict` (함수)**
  - **문제점**: 여러 자산에 대해 `ensemble_predict`를 순차적으로 호출합니다.
  - **개선안**: 모델의 배치 예측(Batch Prediction) 기능을 활용하면 속도를 획기적으로 높일 수 있습니다. (예: 100개 자산을 한 번에 예측)

## 5. 🖥️ `alpha/gui.py`

### 📄 파일 개요
PySide6(Qt)를 사용한 클라이언트 데스크톱 애플리케이션입니다.

### 🔍 주요 코드 분석

- **`WorkerThread` (클래스)**
  - **장점**: UI 프리징(멈춤) 현상을 방지하기 위해 네트워크 요청 등 오래 걸리는 작업을 별도 스레드(QThread)로 분리한 점은 매우 훌륭합니다.
  - **코드**: `signals.progress.emit(...)` 등을 통해 메인 스레드와 안전하게 통신합니다.

- **`check_progress` (함수)**
  - **코드**: `try: ... except: pass`
  - **문제점**: **에러 은폐(Error Swallowing)**. 예외가 발생했을 때 아무런 처리를 하지 않고 넘어갑니다. 디버깅을 어렵게 만들고, 심각한 오류를 놓치게 할 수 있습니다.
  - **개선안**: 최소한 `print(e)`나 로깅을 하거나, UI 상태바에 "연결 재시도 중..." 같은 메시지를 표시해야 합니다.
  - **방식**: `QTimer`를 사용한 폴링(Polling) 방식은 서버에 부하를 줄 수 있습니다. 가능하다면 웹소켓(WebSocket) 도입이 이상적입니다.
